## 3 практическая

**Кратко:**
Работа с сетью, REST API, подключение Retrofit, получение данных и отображение их в списке с помощью RecyclerView.

Суть задания:

### 1 — Выбор REST API

Вам необходимо выбрать любое публичное REST API, которое возвращает список данных и не требует сложной авторизации. Это может быть список постов, пользователей, товаров, изображений и т.д.

[пример раз](https://github.com/alexwohlbruck/cat-facts/tree/master)
[пример два](https://github.com/davemachado/public-api)

Перед началом разработки обязательно откройте выбранный эндпоинт в браузере или Postman и посмотрите, как выглядит JSON-ответ. Это важно, так как под него вы будете создавать DTO-модели.

### 2 — Подключение Retrofit

В проект необходимо подключить библиотеку Retrofit и конвертер для работы с JSON (Gson или Moshi на выбор).

После подключения создаётся интерфейс API, в котором описывается HTTP-запрос. На этом этапе достаточно одного запроса, возвращающего список данных. Архитектуру усложнять не нужно — цель практики в том, чтобы понять саму механику работы с сетью.

### 3 — Создание DTO

На основе JSON-ответа создаются DTO-классы. Они должны соответствовать структуре данных, приходящих с сервера.

DTO используются только для работы с сетью и не содержат логики. Если какие-то поля могут быть `null`, это необходимо явно учитывать, чтобы приложение не падало при парсинге данных.

### 4 — Асинхронный сетевой запрос

Сетевой запрос **не должен выполняться в главном потоке**.

Запрос оформляется как `suspend`-функция и вызывается в фоне с использованием coroutines. Если запрос завершился успешно, данные сохраняются и передаются в UI. Если произошла ошибка, приложение должно корректно её обработать.

Минимальная обработка ошибок обязательна, но сложные сценарии пока не требуются.

### 5 — Отображение данных в RecyclerView

Для отображения полученных данных необходимо использовать RecyclerView.

Создаётся адаптер, который принимает список данных и отображает их на экране. Разметка элемента списка может быть простой, главное — чтобы было понятно, какие данные пришли с сервера.

После получения данных список должен обновляться и отображаться пользователю.

### 6 — Состояния экрана

Во время загрузки данных приложение должно показывать пользователю, что происходит. Это может быть текст «Загрузка…» или любой другой простой индикатор.

Если запрос завершился ошибкой, пользователь должен увидеть сообщение об ошибке, а не пустой экран или падение приложения.

### 7 — Структура проекта

В проекте должна начать формироваться базовая структура:

* код для работы с сетью
* DTO-модели
* Activity и RecyclerView Adapter

Repository, DI и MVVM **на этом этапе не используются**, они будут добавлены в следующих практических работах.

### Критерии приёмки

* Используется Retrofit для работы с REST API
* Данные загружаются из реального публичного API
* Сетевой запрос выполняется асинхронно
* Полученные данные отображаются в RecyclerView
* Приложение корректно обрабатывает ошибку сети
* Проект продолжает использовать XML, без Jetpack Compose
