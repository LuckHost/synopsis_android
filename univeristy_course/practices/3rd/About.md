## 3 практическая

**Кратко:**
Переход на MVVM, перенос состояния экрана в ViewModel и разделение UI и логики.

Хороший гайд для прохождения этой лабораторной находится [здесь](https://youtu.be/KeQWIu8bA-Y?si=qBa1iE8z7_R-YfOF)

Суть задания:

#### 1 — Зачем нужен MVVM

Разделяем логику приложения от представления. UI должен говорить **как и когда** показать, логика приложения - **что и где**

Сделано это для того, чтобы без боли редактировать UI, когда мы хотим, не касаясь при этом логики приложения и наоборот.

Это называется *независимость*
![](./images/independence.jpg)

#### 2 — Создание ViewModel

[вот тут очень хорошо рассказано показано](https://developer.android.com/topic/libraries/architecture/viewmodel?hl=ru)

ViewModel **не должна знать о UI**. Она не знает, существует ли Activity, какие там кнопки и как именно отображаются данные.

**Не используем Jetpack Compose, он будет дальше по курсу, сейчас работаетм с XML**

#### 3 — Перенос логики из Activity

Вся логика, связанная с данными, кешированием и обработкой ошибок, должна быть в ViewModel.

Пока что у нас имеются только хардкодные данные, именно их и храним в ViewModel

В будущем мы будем работать с полученной из интернета информацией именно здесь.

#### 4 — Обновление UI

UI должен реагировать только на изменения состояния, приходящие из ViewModel.

RecyclerView обновляется при изменении данных, индикаторы загрузки и ошибки показываются на основе текущего состояния.

UI не принимает решений и не содержит логики.

### Критерии приёмки

* Используется паттерн MVVM
* Вся логика экрана вынесена в ViewModel
* Activity не содержит бизнес-логики
* ViewModel не знает о UI
* git история коммитов показывает процесс работы
